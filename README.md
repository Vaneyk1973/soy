## Запуск сервиса
### Локально (Python)
```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn app:app --host 0.0.0.0 --port 8000
```
Проверка:
```bash
curl -i http://localhost:8000/health
```
Ожидаемый ответ: `200 OK` и тело `{"status":"ok"}`.

### В Docker
```bash
docker compose up --build
```
Проверка:
```bash
curl -i http://localhost:8000/health
```

## C4 Container диаграмма
Файл: `c4-container.md`

## Домены и ответственность
- **Feed**: формирование персонализированной ленты (ранжирование, признаки, фичи).
- **Catalog**: товары, категории, атрибуты, остатки, цены, поисковые индексы.
- **User**: пользователи, роли, продавцы, профили.
- **Order**: корзина/чекаут, оформление заказа, статусы, отмены.
- **Payment**: платежные намерения, списания, возвраты, учет (ledger).
- **Notification**: уведомления о статусах заказов (email/SMS/push).

## Распределение доменов по сервисам
- `Feed Service` — домен Feed.
- `Catalog Service` — домен Catalog.
- `User Service` — домен User.
- `Order Service` — домен Order.
- `Payment Service` — домен Payment.
- `Notification Service` — домен Notification.
- `API Gateway` — единая точка входа, routing, auth, rate limiting.

**Логика разбиения:** домены с разными темпами изменений и данными разделены по сервисам, чтобы уменьшить связанность и упростить автономное развитие.

## Границы владения данными
- **Feed Service** владеет данными фич и моделей персонализации (Feature Store, профили интересов).
- **Catalog Service** владеет товарами, атрибутами, остатками, ценами.
- **User Service** владеет профилями, учетными данными, ролями.
- **Order Service** владеет заказами и их статусами.
- **Payment Service** владеет платежами и учетными проводками.
- **Notification Service** владеет журналом и статусами уведомлений.

**Общее правило:** нет разделяемых БД между сервисами.

## Взаимодействия сервисов
- **Синхронные** (REST/gRPC):
  - `API Gateway` → все доменные сервисы.
  - `Order Service` → `Payment Service` (создание платежа/возврата).
- **Асинхронные** (Message Bus):
  - `Order Service` → `Notification Service` (события заказа).
  - `Payment Service` → `Notification Service` (события оплаты).
  - `Catalog Service` → `Feed Service` (обновления каталога/остатков).

## Персонализация (кратко)
Используется смесь правил (контекст, категории, новинки) и ML-моделей ранжирования. Данные для персонализации хранятся в `Feed Service` и обновляются асинхронно.

## Альтернативные варианты декомпозиции
### Вариант A: «Минимальный набор сервисов»
- `User+Catalog+Feed` объединены в один сервис.
- `Order+Payment` объединены.
- `Notification` отдельным сервисом.

**Плюсы:** меньше сервисов, проще сопровождение, дешевле инфраструктура.  
**Минусы:** сильная связность, риск блокировок релизов, сложнее масштабирование.

### Вариант B: «Функционально независимые сервисы» (выбранный)
- Каждый домен — отдельный сервис.
- Общение через API Gateway и шину событий.

**Плюсы:** независимые релизы, четкие границы данных, проще масштабирование.  
**Минусы:** больше операционных затрат, сложнее наблюдаемость.

## Финальный выбор
Выбран **Вариант B**, потому что:
- Требования кейса подразумевают явные домены и владение данными.
- Нужна гибкость и масштабирование отдельных частей (каталог/лента/заказы).
- Упрощается соблюдение запрета на shared DB.
